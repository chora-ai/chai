- [ ] How to make system context more customizable? The agents/worker context provided to the orchestrator and the skills context provided to the orchestrator and worker agents should be customizable without updating the source code. Either the source code would still have a backup or the files that allow a user to update the context would be included in the default profiles. There would be an orchestrator context and a skills context, most likely in the root of the profile (not specific to agents but rather what is loaded for each orchestrator - the orchestrator context, one orchestrator for now - and each agent - the skills context, all agents with skills enabled). The alternative would be to add the files to each agent profile directory, role worker would ignore an orchestrator context file if it existed, each agent would ignore the skill context file if no skills were loaded. Listing the workers and listing the skills would still need to be part of the system and could replace a special line in the file, e.g. `<list workers>` and `<list skills>`, or the system could simply append the lists to each customizable context. Also, markdown table format should be considered for the workers and skills list and the pros and cons of this approach should be weighed against the pros and cons of the current layout (sections using heading 4 for the worker or skill). The system should not add additional text or the bare minimum so that the ability to customize how the workers and skills are interpreted remains in the customizable context files. This would potentially eliminate the need for the WORKERS_ENABLED and SKILLS_ENABLED system context (leaving only TODAYS_DATE). These were created for conditions in the context that the user writes, e.g. "if skills enabled, follow these guidelines...", and it would reduce the extra inference required by the agents. This should also be designed in a way that is forward thinking, i.e. what additional context may be needed and should be customizable when introducing new features such as chron jobs or chron-like loops?

The **binary-mediated writes** vs **sandbox-aligned primitives** should be captured, maybe in an ADR, before the skills are removed from the bundled skills. They can also be documented as examples while not being implemented as part of the bundled skills.
